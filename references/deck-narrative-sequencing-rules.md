# Deck Narrative Sequencing Rules

Use these rules during deck planning, before slide-level class, grammar, or variant decisions.

The goal is to make the deck coherent at the section level before optimizing individual slides.

## Purpose

The deck should not be planned as a loose list of good-looking slides.

It should be planned as a narrative spine with ordered sections that each do a different job.

For business, marketing, finance, and case-competition decks, the default section spine is:

1. `Executive Summary`
2. `Analysis`
3. `Last Analysis`
4. `Overview Strategy`
5. `Strategy`
6. `Implementation`
7. `Impact`
8. `Risk & Mitigation`

This spine controls section order, not fixed slide count.

Slide count remains adaptive.

## Section Roles

### 1. Executive Summary

Purpose:

- show the answer early
- frame the challenge
- preview the strategic moves
- preview the impact

This section should feel synthesis-led, not evidence-heavy.

### 2. Analysis

Purpose:

- build the case with evidence
- explain the market, customer, competition, diagnosis, or structural problem
- narrow the problem space

This section can contain multiple slides.

### 3. Last Analysis

Purpose:

- close the analytical argument
- compress the strongest implications
- lead directly into recommendation

This is not just another analysis page.

This section may be one slide or, when needed, a very small cluster.

Its last page should trigger `analysis_terminal_role = bridge-to-recommendation`.

### 4. Overview Strategy

Purpose:

- present the overall answer in one strategic frame
- summarize the strategic architecture before diving into parts

This is usually higher-level than the detailed strategy pages that follow.

### 5. Strategy

Purpose:

- explain the chosen strategic moves in more detail
- show pillars, choices, logic, or design of the recommendation

This section can contain several slides.

### 6. Implementation

Purpose:

- explain how the strategy gets executed
- show roadmap, workstreams, operating model, sequencing, or governance

### 7. Impact

Purpose:

- prove expected business, operational, financial, or customer impact
- show KPI logic, business case, ROI, payback, or expected outcomes

### 8. Risk & Mitigation

Purpose:

- show what could fail
- show how the plan handles uncertainty, constraints, or adoption risk

This section should usually sit near the end of the main deck, after the main strategy and impact are clear.

## Adaptive Slide Allocation

Do not assign the same number of slides to every section.

Instead, allocate adaptively based on source density and argument load.

Typical behavior:

- `Executive Summary`
  - usually compact
- `Analysis`
  - often the largest section
- `Last Analysis`
  - usually compact but strategically important
- `Overview Strategy`
  - compact
- `Strategy`
  - expands when the recommendation has several pillars or mechanisms
- `Implementation`
  - expands when execution complexity is high
- `Impact`
  - expands when financial or KPI proof is central
- `Risk & Mitigation`
  - usually compact unless the case explicitly centers risk

## Planning Order

During deck planning, use this order:

1. identify the deck's likely section spine from the source
2. map source content blocks into the section spine
3. decide which sections are mandatory, optional, merged, or expanded
4. allocate adaptive slide counts inside each section
5. mark the final page of the analysis section as `Last Analysis`
6. only then create slide-by-slide purposes and messages

## Merge and Omit Rules

The spine is a default, not a prison.

Allowed adjustments:

- merge `Overview Strategy` into `Executive Summary` when the deck is very short
- merge `Impact` and `Risk & Mitigation` when the source is sparse
- merge `Implementation` into `Strategy` when execution detail is light
- omit a section only when the source truly does not support it

Do not:

- skip `Last Analysis` when the deck still needs an analytical bridge into recommendation
- let `Strategy` appear before the deck has finished its analytical case
- let `Impact` appear before the recommendation is clear

## Output Expectation

At the end of deck planning, the planner should know:

- section order
- which slides belong to each section
- which slide is the last analysis bridge
- where recommendation begins

Only after that should slide-level body planning begin.
