# Implementation Modes

Use this file when a slide is primarily about:

- `implementation`
- `rollout`
- `workstreams`
- `milestones`
- `governance`
- `owners`
- `dependencies`

Implementation pages are not all the same. Decide the dominant execution proof first, then choose grammar and variant.

## Purpose

Implementation mode answers:

- what kind of execution proof this page is really carrying
- whether time, control points, or governance should dominate the page
- which implementation grammar family should lead the composition

## Modes

### `timeline-led`

Use when the page's main job is to show:

- phased rollout
- sequence across years or stages
- workstream timing
- milestone rhythm

Typical signs:

- the audience must understand `when` each initiative happens
- year markers or phase names are the first thing that should organize the page
- execution tracks sit beneath a clear time backbone

Do not use when:

- the real message is checkpoint control rather than elapsed time
- owners, dependencies, or control structure matter more than the timeline itself

### `decision-gate-led`

Use when the page's main job is to show:

- pilot first, scale later
- go / no-go checkpoints
- evaluation moments between phases
- activation that depends on proof of success

Typical signs:

- the roadmap should not read as one uninterrupted bar
- checkpoints are part of the argument, not decoration
- the audience must see why later phases are conditional

Do not use when:

- the slide is just a simple phased rollout
- the page has no meaningful evaluation or gate logic

### `governance-led`

Use when the page's main job is to show:

- owner structure
- execution control
- dependencies
- coordination logic
- control tower or orchestration behavior

Typical signs:

- the critical question is `who drives what` or `how execution is controlled`
- time is still relevant but is not the main proof
- the page needs a control anchor, dependency strip, or owner structure

Do not use when:

- the audience mainly needs to see a time-based rollout
- the page can already be understood through a simple phased roadmap

## Selection Rule

Pick exactly one implementation mode per implementation-heavy slide:

1. If the main proof is `when things happen`, choose `timeline-led`.
2. If the main proof is `when scale is allowed to proceed`, choose `decision-gate-led`.
3. If the main proof is `how execution is controlled`, choose `governance-led`.

## Non-Negotiables

- Do not default every implementation page to `timeline-led`.
- Do not choose `decision-gate-led` unless the gates actually matter to the story.
- Do not choose `governance-led` unless owners, dependencies, or control logic are central.
- Do not let a secondary detail band determine the mode; choose based on the dominant execution proof.
